- Освоить стратегии инвалидации кеша при изменении данных (Create, Update, Delete).

- Закрепить модульную архитектуру приложения, добавив слой кеширования без нарушения принципов инкапсуляции.

- Требования к использованию библиотек кеширования:
  В рамках данной работы допускается использование сторонних решений для взаимодействия с Redis, однако с целью закрепления понимания принципов кеширования вводятся следующие ограничения:
    - Разрешено использование низкоуровневых клиентов Redis (драйверов) для вашего языка программирования (например, `ioredis`, `redis-py`, `go-redis`, `predis`, `phpredis` и т.п.).
    - Разрешено использование высокоуровневых абстракций кеширования фреймворка (например, `@nestjs/cache-manager`, `Spring Cache`, `Django Cache Framework`, `Laravel Cache`), только если они предоставляют возможность явного управления ключами, TTL и ручной инвалидации через сервисный слой.
    - Запрещено использование библиотек автоматического кеширования (Auto-Caching), которые перехватывают запросы на уровне роутинга/контроллера и не позволяют контролировать содержимое кеша и стратегию инвалидации в коде сервиса.

  Логика формирования ключей (префиксы, структура) и логика инвалидации (удаление при записи) должны быть реализованы явно в коде сервиса, а не скрыты исключительно в конфигурационных файлах.

## Краткие теоретические сведения

### 1. Кеширование
Кеширование — это механизм хранения копии данных в быстром хранилище (обычно в оперативной памяти) для ускорения последующих запросов. В веб-приложениях это снижает нагрузку на базу данных (БД) и уменьшает время отклика (Latency).
- Cache-Aside (Lazy Loading): Приложение сначала проверяет кеш. Если данных нет (Cache Miss), они загружаются из БД, сохраняются в кеш и возвращаются клиенту.
- Инвалидация кеша: Процесс удаления или обновления устаревших данных в кеше при изменении данных в БД (CREATE, UPDATE, DELETE). Без инвалидации клиенты могут получать неактуальную информацию.

### 2. Redis
Redis (Remote Dictionary Server) — это хранилище типа «ключ-значение» (Key-Value), работающее в памяти.
- TTL (Time To Live): Время жизни ключа. Позволяет автоматически удалять данные через заданный интервал, предотвращая переполнение памяти и устаревание данных.
- Структуры данных: Строки, Хешы, Списки, Множества. Для кеширования ответов API чаще всего используются строки (сериализованный JSON).
- Безопасность: Redis должен быть защищен паролем и изолирован от публичного доступа, так как по умолчанию не шифрует трафик.

### 3. Безопасность и сессии
- Токены доступа (Access Tokens): Обычно JWT токены проверяются Stateless (без обращения к БД). Однако для реализации мгновенного выхода (Logout) или отзыва доступа необходимо хранить идентификаторы токенов (JTI) или их хеши в Redis с TTL, равным времени жизни токена. Это позволяет создать «черный список» или реестр активных сессий.
- Токены обновления (Refresh Tokens): В Лабораторной работе №3 они хранились в БД. Для ускорения проверки валидности сессии их идентификаторы также можно кешировать в Redis.
- Запрещенные данные: В кеш категорически запрещено помещать пароли (даже хеши) и чувствительные персональные данные без необходимости.

### 4. Именование ключей
Для удобства управления и предотвращения конфликтов рекомендуется использовать иерархические префиксы:
- Формат: `app:module:entity:identifier`
- Пример: `wp:items:list:page:1`, `wp:auth:user:{userId}:access:jti:{jti}`

## Ход работы

В рамках данной работы необходимо интегрировать систему кеширования в существующее приложение, реализующее CRUD и авторизацию.

### 1. Подготовка инфраструктуры (Docker)
Обновите конфигурацию `docker-compose.yml`, добавив сервис Redis. Обязательно настройте требование пароля для подключения. Убедитесь, что сервис приложения имеет доступ к сети Redis и переменные окружения настроены корректно.

Пример обновленного `docker-compose.yml`:
```yaml
version: "3.8"

services:
  postgres:
    image: postgres:16
    container_name: wp_labs_db
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - wp_labs_postgres:/var/lib/postgresql/data
    networks:
      - wp_labs_network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: wp_labs_redis
    restart: unless-stopped
    ports:
      - "6379:6379"
    volumes:
      - wp_labs_redis:/data
    networks:
      - wp_labs_network
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    healthcheck:
      test: ["CMD", "redis-cli", "--pass", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    
  #app:
    ### ...

volumes:
  wp_labs_postgres:
  wp_labs_redis:

networks:
  wp_labs_network:
```

Обновите `.env` файл:
```

REDIS_HOST=redis
REDIS_PORT=6379
REDIS_PASSWORD=redis_secure_password_change_in_prod
CACHE_TTL_DEFAULT=300

```

### 2. Модуль кеширования
Создайте отдельный сервис или модуль для работы с Redis. Не разбрасывайте логику подключения по всему коду.
- Реализуйте методы: `get(key)`, `set(key, value, ttl)`, `del(key)`, `delByPattern(pattern)`.
- Используйте сериализацию (JSON) для хранения объектов.
- Настройте клиент Redis на использование пароля из переменных окружения.
- Обработайте случаи, когда Redis недоступен (приложение не должно падать, кеш должен быть опциональным).

Пример структуры модуля:
```text
src/
  common/
    cache/
      cache.module.ts (или аналог для вашего языка)
      cache.service.ts
  items/
    items.service.ts  // Бизнес-логика с кешированием
  auth/
    auth.service.ts   // Логика токенов с Redis
```

### 3. Кеширование списков ресурсов (GET /items)
Модифицируйте сервис ресурсов (из ЛР №2).
- При запросе `GET /items?page=1&limit=10`:
    1.  Сформируйте уникальный ключ кеша, включающий параметры пагинации.
    2.  Проверьте наличие в кеше.
    3.  Если есть (Hit) — верните данные.
    4.  Если нет (Miss) — запросите из БД, сохраните в кеш с TTL, верните ответ.
- Инвалидация:
    - При `POST /items` (создание) — очистите ключи кеша списков.
    - При `DELETE /items/:id` (удаление) — очистите ключи кеша списков и ключ конкретного элемента.

### 4. Управление Access токенами в Redis
В Лабораторной работе №3 вы реализовали хранение Refresh токенов в БД. Для доступа добавьте слой Redis:
- Хранение JTI: При генерации Access токена извлекайте его уникальный идентификатор (JTI - JWT ID).
- Запись в Redis: Сохраните ключ `wp:auth:user:{userId}:access:{jti}` со значением `valid` (или userId) и TTL, равным времени жизни Access токена (например, 15 минут).
- Проверка: В Guard/Middleware после проверки подписи JWT проверяйте наличие JTI в Redis. Если ключа нет (удален вручную) — доступ запрещен (401).
- Инвалидация: При `/auth/logout` удаляйте ключ JTI из Redis. Это позволит отозвать доступ до истечения 15 минут.

### 5. Стандарты именования ключей
Используйте иерархическую структуру ключей для удобства управления и безопасности:
- Формат: `app_name:module:entity:identifier`
- Примеры:
    - `wp:items:list:page:1:limit:10`
    - `wp:users:profile:{userId}`
    - `wp:auth:user:{userId}:access:{jti}`
    - `wp:auth:user:{userId}:refresh:{tokenId}`
- Избегайте ключей без префиксов, чтобы предотвратить конфликты имен.

### 6. Работа с Redis CLI (Практическая проверка)
Вам необходимо убедиться, что данные действительно попадают в кеш и удаляются при инвалидации. Обратите внимание на аутентификацию.

Подключитесь к контейнеру Redis:
```bash
docker exec -it wp_labs_redis redis-cli --pass ${REDIS_PASSWORD}
```
Если внутри контейнера переменная окружения не доступна, подставьте пароль вручную.

Полезные команды для проверки:
1.  Просмотр ключей по паттерну:
    ```bash
    KEYS 'wp:*'
    ```
2.  Получение значения ключа:
    ```bash
    GET wp:items:list:page:1:limit:10
    ```
3.  Проверка времени жизни ключа (TTL):
    ```bash
    TTL wp:items:list:page:1:limit:10
    ```
4.  Удаление ключа (ручная инвалидация):
    ```bash
    DEL wp:items:list:page:1:limit:10
    ```
5.  Удаление по паттерну (массовая инвалидация):
    ```bash
    # В redis-cli версии 6+ можно использовать UNLINK для асинхронного удаления
    UNLINK wp:items:* 
    ```
6.  Очистка всей базы (для тестов):
    ```bash
    FLUSHDB
    ```

### 7. Эндпоинты и поведение
| Метод | URI | Действие с кешем |
| :--- | :--- | :--- |
| GET | `/items` | Проверка кеша. При отсутствии кеша выполните запись в кеш. |
| POST | `/items` | Выполните инвалидацию ключей списков `/items`. |
| PUT/PATCH | `/items/:id` | Выполните инвалидацию ключа списка и ключа конкретного элемента. |
| DELETE | `/items/:id` | Выполните инвалидацию ключа списка и ключа элемента. |
| GET | `/auth/whoami` | Проверка кеша профиля пользователя. При отсутствии кеша выполните запись в кеш. |
| POST | `/auth/logout` | Инвалидация кеша сессии (JTI) и профиля. |

### 8. Тестирование
1. Запустите приложение: `docker-compose up --build`.
2. Создайте несколько записей через API.
3. Выполните `GET /items`. Через `redis-cli` убедитесь, что появился ключ кеша.
4. Выполните `GET /items` повторно. Время ответа должно быть быстрее, логи приложения не должны содержать запросов к БД, если реализовано логгирование SQL.
5. Выполните `POST /items`. Через `redis-cli` убедитесь, что ключи кеша списков удалены.
6. Войдите в систему, получите Access токен. Проверьте наличие ключа `wp:auth:user:{userId}:access:{jti}` в Redis.
7. Выполните `/auth/logout`. Убедитесь, что ключ токена удален из Redis, и повторный запрос с этим токеном возвращает 401.
